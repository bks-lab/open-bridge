# Git Hygiene — mechanics gates for Bridge repos

Three git mechanics that bite *before* the work is even reviewed: a missing
sign-off turns the PR red, a non-atomic stage hands your files to a parallel
agent, and the wrong path makes `git add` fail outright. Get them right from the
first commit. For *what may land on which upstream* see
[`rules/operations.md` § Commit Hygiene](operations.md#commit-hygiene); for the
content leak-scan before a promote see [`rules/promote-safety.md`](promote-safety.md).

## 1. DCO sign-off — always `git commit -s`

All three Bridge tier-repos (your personal instance `<you>/<your-bridge>`, an
org overlay like `<your-org>/<your-org>-bridge`, and the OSS upstream
`bks-lab/open-bridge`) run a `DCO` workflow (`.github/workflows/dco.yml`) that
checks **every** commit in the PR range for a `Signed-off-by: Name <email>`
trailer. A commit without it fails the "Verify Developer Certificate of Origin"
check — separate from the YAML/frontmatter `Validate` pipeline, so it goes red on
its own.

- **Always `git commit -s` (`--signoff`) from the first commit.** The trailer is
  generated from `git config user.name` / `user.email`.
- **Fixing already-pushed unsigned commits is a whole round-trip:**
  `git rebase --signoff <base>` (e.g. `origin/development` / `origin/main`) then
  a **force-push** — and force-push is gated (see gate 2 + SOUL "pause before
  `push --force`"), needing explicit user sign-off. Avoid the loop by signing
  from the start.
- `dco.yml` triggers only on `pull_request`, not on a plain branch push.
- The DCO `Signed-off-by` trailer is the repo's own contributor attestation, **not**
  AI attribution — it does **not** fall under the Co-Authored-By / "Generated by"
  prohibition.

## 2. Atomic stage + commit — one shell call, explicit paths

A parallel autopilot agent sometimes runs on the same working copy and commits +
pushes to the user branch on its own. The git index is **shared state** between
both agents, so any gap between your `git add` and your `git commit` is a race
window in which the autopilot can sweep your staged files into *its* commit
(observed: CORE files landed in a foreign commit with the wrong title and a mixed
CORE/USER scope, already pushed).

- **`git add <explicit paths> && git commit -m …` in ONE shell call** (or
  `git commit <pathspec> -m …` with no separate `add`). Never split staging and
  committing across two calls.
- **Stage only the intended paths.** Never `git add -A` / `git add .` — it grabs
  whatever the autopilot left in the tree.
- **Never force-fix already-pushed foreign history** while origin is still moving.
  The autopilot's `git pull --rebase` will orphan a clean rewrite, and
  `--force-with-lease` keeps failing because origin no longer sits at the expected
  hash. Once a messy commit is buried under upstream work, leave it — the content
  is correct, the cosmetic scope-mix is not worth the rewrite. If a rewrite is
  truly required, first confirm the autopilot is off (origin stable for several
  seconds), tag a safety point (`git tag backup-pre-cleanup <tip>`), and use
  `--force-with-lease` with the explicit expected hash.

## 3. `skills/` symlink — stage the real path

`.claude/skills` is a **symlink** to `skills/`. Edit and Read through
`.claude/skills/<skill>/…` work (same inode, changes land in `skills/…`), but
`git add` does not follow the symlink:

- `git add .claude/skills/…` → `fatal: pathspec '…' is beyond a symbolic link`.
- **Always stage skill files via the real tracked path** `skills/<skill>/…`.
  `git ls-files | grep <skill>` shows the canonical path. Applies to every commit
  that touches a skill.
